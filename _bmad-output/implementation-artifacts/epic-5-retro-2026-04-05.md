## 3. What Went Well

### 3.1 PM-ET Delivered as a Complete, Validated Module

`src/climate_indices/pm_eto.py` (465 lines) implements the full FAO-56 ETo pipeline end-to-end:
atmospheric helpers, vapor pressure helpers, humidity pathway dispatcher, and core Equation 6.
The 102 tests in `test_pm_eto.py` validate against both the Bangkok tropical and Uccle temperate
FAO-56 worked examples within ±0.05 mm/day. This is a research-grade deliverable — not a stub.

### 3.2 EDDI Wired into the Public API

EDDI went from an algorithm in `indices.py` to a fully first-class public index:
- `@overload` signatures in `typed_public_api.py` for NumPy and xarray dispatch
- CF metadata entry (`dimensionless`, Hobbins et al. 2016 full citation)
- CLI integration via `--index eddi`
- NOAA reference validation infrastructure (3 self-consistency tests always pass; 9 reference
  tests skip gracefully when fixtures absent — documented as acceptable for v2.4.0)
- Exported from `climate_indices.__all__`

### 3.3 42/42 Pattern Compliance — All 7 Indices, All 6 Patterns

The compliance dashboard (`test_pattern_compliance.py`) validates 42 points via AST introspection
(no import side effects). Achieving 100% coverage across SPI, SPEI, PET Thornthwaite,
PET Hargreaves, PNP, PCI, and Palmer is the architectural cohesion goal of the entire v2.4.0
sprint — and it landed.

### 3.4 `mypy --strict` Clean Across 16 Modules

Reaching strict mypy compliance on a library with pre-existing technical debt required
targeted fixes: 4 `no-any-return` suppressions in `pm_eto.py` and 2 explicit `cast()` calls
in `indices.py`. The result is a type-safe public API that IDEs can reason about fully.

### 3.5 Senior Review Caught Real Issues Pre-Release

The Story 5.3 senior review (James, Opus 4.6) identified 4 Medium and 2 Low issues,
all fixed before release:
- CF metadata `units` field was `""` (empty string) instead of `"dimensionless"` — an
  inconsistency against SPI/SPEI that would have been confusing to downstream users
- References were abbreviated; expanded to full academic citations with DOIs/publishers
- 15 regression tests added for the 5 new CF metadata entries
- Dead `if TYPE_CHECKING: pass` block and stale module docstring cleaned up

### 3.6 Clean Release Cut

`pyproject.toml` at `2.4.0`, CHANGELOG `[Unreleased]` section replaced with
`[2.4.0] - 2026-04-05`, Palmer xarray deferral documented with explicit v2.5.0 attribution.
No ambiguity about what shipped and what didn't.

## 4. Challenges and Growth Areas

### 4.1 Story 5.1 Integration Complexity Underestimated

**What happened:** The compliance dashboard dry-run at the start of Story 5.1 showed 31/80
assertions failing — nearly 40% of the compliance matrix broken. Root cause: Epic 1
implementation commits had not been merged from `feature/epic-1-canonical-patterns` into the
integration branch. Nine cherry-picks (oldest-to-newest) were required to resolve it, including
one conflict in `test_property_based.py`.

**Systemic issue:** Epic branches were developed in isolation and the integration branch was
never kept current. Story 5.1's charter was listed as "audit run" but turned into a
non-trivial integration task.

**Impact:** Consumed a significant portion of Story 5.1 effort. The development team had to
also commit pre-existing unstaged `exceptions.py` changes mid-audit, which added noise to
the commit graph.

### 4.2 Epic 4 Scope Drift — `palmer_xarray()` Marked Done but Not Committed

**What happened:** Epic 4 stories 4.3 through 4.9 were marked `done` in sprint-status.yaml,
covering the `palmer_xarray()` manual wrapper, multi-output Dataset return, AWC handling, and
typed API overloads. Story 5.3's investigation found zero commits implementing any of this —
`palmer_xarray` has no matches in `palmer.py`. All 7 Palmer FRs (FR-PALMER-001 through
FR-PALMER-007) remain undelivered.

**Systemic issue:** Sprint status was updated to `done` without the corresponding code
being committed. There was no gate checking "does the committed code match the story's
acceptance criteria?" before marking a story done.

**Impact:** 7 FRs deferred to v2.5.0. Users of the Palmer module who expected a multi-output
xarray interface will not find one in v2.4.0. The deferral is documented in CHANGELOG and a
`TODO(v2.5.0)` comment is at the end of `palmer.py`.

### 4.3 Story 5.2 Dev Record Not Updated

**What happened:** Story 5.2 (`stories/5-2-reference-validation-final-check.md`) has an empty
Dev Agent Record — no debug log, no completion notes, no file list — and all task checkboxes
remain unchecked, despite the story being marked `done` in sprint-status and the related
commits being present in git history.

**Systemic issue:** The story artifact was not kept in sync with the actual development
activity. The sprint-status entry was the source of truth, but the story file became stale.

**Impact:** Low code impact (the EDDI reference validation infrastructure is in git). High
traceability impact — future developers reviewing this story cannot understand what decisions
were made or why.

### 4.4 AC Specification Drift (Story 5.1, AC #4)

**What happened:** Acceptance criterion #4 specified adding a compliance step to
`.github/workflows/ci.yml`. That file does not exist in the repository — the active CI
workflow is `unit-tests-workflow.yml`. The step was added to the correct file, but the
mismatch required in-flight deviation from a written AC.

**Systemic issue:** ACs referenced an artifact (ci.yml) that didn't exist, without a prior
audit of the actual CI configuration.

## 5. Key Insights and Lessons Learned

### L1: Integration Branch Needs Continuous Merge — Not One-Shot at Validation Gate

Running cherry-picks at the end of the sprint created unnecessary risk and effort. Keeping the
integration branch current as each epic branch completes would have made Story 5.1 a 30-minute
smoke test rather than a rescue operation.

**Apply in v2.5.0:** Merge epic branches into the integration/feature branch within one sprint
of completion — do not allow branches to age for months.

### L2: Story "Done" Must Be Backed by Committed Code, Not Just sprint-status

The Epic 4 scope drift happened because `sprint-status.yaml` was treated as the truth signal
for story completion, while the actual code was never committed. The canonical truth is the
git history, not the YAML.

**Apply in v2.5.0:** Before marking a story `done` in sprint-status, verify with
`git log --oneline` that the implementing commits exist on the target branch. If they don't,
the story is not done.

### L3: Story Artifacts Are Traceability Records, Not Just Task Lists

Story files exist to let future developers (or future Claude sessions) reconstruct decisions,
alternatives considered, and what was hard. Story 5.2's empty dev record means that context
is lost. The 20 minutes it takes to write completion notes pays dividends in every future
session that touches that code.

**Apply in v2.5.0:** Dev Agent Record must be populated — at minimum, Completion Notes List
and File List — before marking a story done.

### L4: Senior Review Scope Should Match Risk, Not Just Story Size

Story 5.3 was small by line count but high-risk (release gating, first-time CF metadata
entries, public API wiring). The Opus 4.6 senior review was proportionate to that risk and
caught real issues. Stories 5.1 and 5.2 did not receive equivalent review depth despite
Story 5.1's integration complexity.

**Apply in v2.5.0:** Use story risk (integration scope, public API surface, scientific
accuracy requirements) as the trigger for senior review depth, not story line count.

### L5: The `palmer_xarray()` Deferral Is Now the Critical Path for v2.5.0

`palmer_xarray()` is the most architecturally significant undelivered item in the entire
v2.4.0 sprint. It is not a simple addition — it requires Pattern C (stack/unpack workaround
for xarray Issue #1815) and returns a `xr.Dataset` with 4 variables, which is structurally
different from every other index in the library. It should be the first story created for
v2.5.0 planning, with dedicated architecture review time before any code is written.

## 9. Significant Discovery: `palmer_xarray()` Scope Drift

Bob (Scrum Master): "This warrants explicit documentation as a significant discovery — it affected 7 FRs."

**Summary:** Epic 4 stories 4.3–4.9 were marked `done` in sprint-status.yaml. Story 5.3 found
zero committed code implementing `palmer_xarray()`. The stories appear to have been marked done
based on design work or local exploration, not committed implementation.

**Wrong assumption in Epic 4 plan:** Stories could be marked done as code was written locally.

**Actual reality:** Git history is the only authoritative record of delivered work.

**Impact on v2.5.0:** The Palmer xarray wrapper is the most architecturally complex item
remaining. It requires:
1. Pattern C implementation (stack/unpack for xarray Issue #1815)
2. `xr.Dataset` return with 4 named variables (pdsi, phdi, pmdi, z_index)
3. `@overload` in `typed_public_api.py` with Dataset return type
4. AWC handling (no time dimension)
5. Equivalence tests at 1e-8 tolerance

**Recommendation:** Palmer xarray should be the **first epic** planned for v2.5.0. Do not
start v2.5.0 with any other work until Palmer xarray architecture is reviewed and a realistic
story breakdown is written. The architecture.md Pattern C section is the primary reference.

**Epic update required:** YES — v2.5.0 must account for Palmer xarray as a first-class epic,
not an "already done" carryover.

## 12. Key Takeaways

1. **Integration debt accumulates silently.** Running 9 cherry-picks at the validation gate was
   avoidable. Keeping the integration branch current as epics complete is cheap; catching up at
   release time is expensive.

2. **sprint-status.yaml is a status tracker, not a truth oracle.** The Epic 4 scope drift
   demonstrates that sprint tracking and actual code delivery can diverge. Git log is always
   more authoritative.

3. **v2.4.0 shipped a genuinely capable release.** PM-ET (FAO-56), EDDI public API, 42/42
   pattern compliance, and mypy --strict across 16 modules represent real, validated, production-
   quality science infrastructure. The Palmer deferral is the exception, not the rule.

4. **The next milestone's critical path starts here.** `palmer_xarray()` using Pattern C is the
   most architecturally novel work ahead. Starting v2.5.0 planning with that work ensures the
   rest of the release isn't blocked by it.
